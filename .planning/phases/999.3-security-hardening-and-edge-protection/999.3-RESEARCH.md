# Phase 999.3: Security Hardening & Edge Protection - Research

## Research Objective

Answer: what needs to be true for this codebase to resist brute-force abuse, reduce app-layer DoS risk, and operate safely behind a production edge?

## Current Codebase Findings

### 1. Authentication is easy to brute-force today

Observed in:
- `elearning-api/src/routes/auth.routes.js`
- `elearning-api/src/controllers/auth.controller.js`
- `elearning-api/src/services/auth.service.js`

Current behavior:
- `/api/auth/login` has no rate limiting.
- The controller accepts raw email/password and calls the service directly.
- The service returns generic wrong-credential messaging, which is good, but there is no attempt counting, cooldown, or lockout.

Implication:
- credential stuffing and brute-force attacks are currently bounded only by infrastructure bandwidth
- repeated password guessing could also create noisy DB load on `user.findUnique` and `bcrypt.compare`

### 2. Upload abuse is possible without authentication

Observed in:
- `elearning-api/src/routes/upload.routes.js`
- `elearning-webapp/src/utils/api.js`

Current behavior:
- `/api/upload` is not protected by auth middleware.
- Frontend usage is currently from admin screens only.
- Multer accepts up to 50 MB, uses memory storage, and relies on MIME allowlisting only.
- Uploads use `upsert: true`, which is a weak default for public-facing file handling.

Implication:
- anonymous clients can consume CPU, memory, and storage
- MIME spoofing is not strongly defended
- a burst of large uploads can amplify application cost and latency

### 3. Request-surface hardening is too permissive

Observed in:
- `elearning-api/src/server.js`

Current behavior:
- `app.use(cors())` allows any origin by default.
- global JSON and urlencoded parsers both allow 50 MB.
- there is no `helmet` baseline.
- `trust proxy` is not configured.

Implication:
- production origins are not constrained
- every route accepts oversized bodies that only a few routes truly need
- rate limiting keyed by IP would be unreliable behind a reverse proxy unless proxy trust is set correctly

### 4. Edge DDoS protection is not yet formalized

Observed in:
- `.planning/HANDOFF-pre-deploy.md`
- `vercel.json`

Current behavior:
- the repo has deploy notes for DigitalOcean hardening, but no formal edge protection runbook
- frontend response headers are set in `vercel.json`, but backend abuse controls are still mostly inside app code

Implication:
- app-level middleware alone will not protect against volumetric floods or layer-7 bot traffic
- production needs an explicit edge story: CDN/WAF/reverse proxy, TLS termination, bot mitigation, and origin shielding

## Recommended Architecture

## Layer 1: Edge / Infrastructure

Primary purpose:
- absorb or block volumetric traffic before it reaches Node

Recommended controls:
- place the API behind a CDN/WAF or managed reverse proxy
- enforce HTTPS and origin shielding
- enable edge rate limits for `/api/auth/login` and upload paths
- add bot or anomaly detection where available
- keep direct origin access restricted so attackers cannot bypass the edge

Why this is separate from app code:
- Express middleware helps with abusive clients that already reached the app
- DDoS protection for bandwidth floods, SYN floods, and large request storms belongs at the network/edge layer

## Layer 2: Express Security Baseline

Recommended changes:
- add `helmet`
- replace open `cors()` with env-driven allowlists
- set `app.set('trust proxy', value)` from env
- reduce global `express.json()` / `express.urlencoded()` limits to small defaults
- keep upload-specific size allowance inside Multer only

Suggested environment contract:
- `ALLOWED_ORIGINS`
- `TRUST_PROXY`
- `API_BODY_LIMIT`
- `URLENCODED_BODY_LIMIT`
- `UPLOAD_MAX_MB`
- `RATE_LIMIT_REDIS_URL`

## Layer 3: Abuse Controls

### Auth limiter

Recommended pattern:
- IP-based rate limit for `/api/auth/login`
- secondary account key on normalized email plus client IP
- temporary lockout after repeated failures
- structured event log on lockout or threshold crossing

Why both keys matter:
- IP-only limits can hurt NATed offices
- email-only limits can be abused against a victim account
- IP plus account reduces both false positives and easy bypasses

### Sensitive-route limiters

Recommended protected groups:
- `/api/upload`
- `/api/admin/dashboard`
- `/api/admin/analytics`
- admin write endpoints such as create/update/delete actions

Recommended store:
- Redis/Valkey or another shared cache in production
- memory store only for local/dev

## Layer 4: Detection and Audit

Recommended events:
- `auth.login.failed`
- `auth.login.locked`
- `auth.login.rate_limited`
- `upload.rejected`
- `upload.rate_limited`
- `admin.route.rate_limited`

Storage options:
- start with structured application logs plus optional DB persistence for high-value events
- if DB persistence is added, create a focused `SecurityEvent` model rather than mixing with product tables

## Layer 5: Recovery and Operations

Operators need:
- a runbook for temporarily tightening thresholds during attack
- a list of env vars and where each is enforced
- a quick way to distinguish edge blocks from app blocks
- alerting or at least searchable logs for suspicious spikes

## Concrete Package / Module Recommendations

Recommended Node packages:
- `helmet`
- `express-rate-limit`
- `rate-limit-redis` plus `redis` if using Redis-backed shared counters
- `file-type` for content sniffing
- `uuid` or `crypto.randomUUID()` for collision-safe upload keys

Recommended internal modules:
- `src/config/security.js` or `src/middleware/security.js` for bootstrap wiring
- `src/middleware/rateLimiters.js` for route-specific policies
- `src/utils/securityEvents.js` for consistent event emission

## Rollout Order

1. Establish baseline middleware and env contract
2. Protect login and upload first
3. Protect expensive admin and analytics routes
4. Add observability and attack-response runbook
5. Add optional second-line controls such as CAPTCHA or MFA only if attack patterns justify it

## What Should Wait

Not first move:
- full auth token model redesign
- MFA for everyone
- RLS adoption
- antivirus scanning on every upload

These are valuable, but they are not the shortest path to reducing current exposure.

## Recommended Verification

- helper tests for limiter policy selection and env parsing
- auth integration tests that verify repeated failed logins trigger 429 or lockout behavior
- upload tests that verify unauthenticated requests are rejected
- smoke test that allowed origins pass and disallowed origins fail in production mode
- operator checklist confirming the API sits behind an edge service before launch

## Research Conclusion

The codebase does not need a full security rewrite first. It needs a disciplined production baseline:
- edge protection for DDoS
- shared-store rate limiting for auth and sensitive routes
- upload authentication and validation
- CORS/body-limit/security-header hardening
- structured detection and runbook support

That is the right scope for Phase 999.3.
