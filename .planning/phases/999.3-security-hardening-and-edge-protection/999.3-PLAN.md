# Phase 999.3: Security Hardening & Edge Protection - PLAN

## Goal

Reduce immediate production abuse risk by adding layered protections against brute-force attacks, route abuse, oversized payloads, and missing edge defenses without breaking the current JWT-based login flow.

## Exit Criteria

- `/api/auth/login` is protected by shared-store rate limiting and temporary lockout behavior
- `/api/upload` requires authenticated admin-panel access and rejects invalid or oversized files safely
- production CORS is allowlist-based instead of open by default
- global parser limits are reduced and route-specific size allowances are explicit
- security headers and proxy-aware IP handling are configured centrally
- suspicious auth, upload, and limiter events are observable
- the deployment runbook documents edge DDoS controls and operator response steps

## Execution Strategy

### Sequence
1. Establish the shared security baseline and env contract
2. Protect authentication against brute-force abuse
3. Lock down upload and expensive admin routes
4. Add observability and incident-response documentation
5. Verify in CI and with focused backend tests

### Delivery Model
- ship the baseline before tuning thresholds
- use environment-driven defaults so dev, staging, and production can diverge safely
- treat edge protection and app protection as complementary, not interchangeable

## Work Breakdown

### Task Group 1: Security Baseline Bootstrap

#### Objective
Create one place where backend security defaults are configured and easy to audit.

#### Tasks
- Add `helmet` to the API dependencies
- Create `src/config/security.js` or `src/middleware/security.js`
- Move CORS to an env-driven allowlist with localhost exceptions only in non-production
- Add `trust proxy` configuration driven by env
- Reduce global `express.json()` and `express.urlencoded()` limits to safer defaults
- Wire a broad default API limiter for non-sensitive routes
- Document required env vars in `.env.example` or equivalent deployment notes

#### Deliverables
- central security bootstrap module
- updated `src/server.js`
- documented env contract for security settings

#### Verification
- backend starts with missing optional security env vars in development
- disallowed production origins receive a CORS failure
- allowed origins still succeed
- global body limit is lower than the current 50 MB baseline

### Task Group 2: Auth Brute-Force Defense

#### Objective
Slow and temporarily block repeated login abuse without breaking the existing frontend auth contract.

#### Tasks
- Add a dedicated limiter for `POST /api/auth/login`
- Normalize login keys by client IP and lowercased email
- Add temporary lockout handling after repeated failures
- Emit structured events for failed login thresholds, rate limits, and lockouts
- Keep the response body generic so user enumeration stays difficult
- Add tests for repeated login failures and reset behavior after success or timeout

#### Deliverables
- auth limiter middleware
- lockout helper/service
- auth route/controller/service updates as needed
- backend tests for brute-force behavior

#### Verification
- repeated failed logins trigger 429 or lockout behavior
- successful login still returns `{ token, user }`
- nonexistent and wrong-password responses remain indistinguishable

### Task Group 3: Upload and Admin Route Hardening

#### Objective
Close the largest unauthenticated abuse path and reduce cost amplification on expensive routes.

#### Tasks
- Protect `/api/upload` with `verifyToken` and `verifyAdminPanelAccess`
- Remove anonymous access to upload processing
- Replace MIME-only validation with MIME plus file-signature checks where supported
- Generate collision-safe filenames and disable overwrite-by-default behavior
- Add a strict upload limiter with route-specific thresholds
- Add moderate rate limits for `/api/admin/dashboard`, `/api/admin/analytics`, and selected admin write endpoints

#### Deliverables
- updated `src/routes/upload.routes.js`
- route-specific limiter module
- any helper utilities needed for file validation and naming

#### Verification
- unauthenticated upload requests fail with 401
- authenticated non-admin upload requests fail with 403
- invalid file signatures are rejected
- repeated upload bursts trigger limiter responses

### Task Group 4: Observability and Security Events

#### Objective
Make suspicious behavior visible enough to debug, alert on, and respond to.

#### Tasks
- Create a consistent security-event logging helper
- Emit events for login failures, lockouts, upload rejections, and limiter hits
- Decide whether events go to structured logs only or also to a Prisma-backed `SecurityEvent` table
- Add correlation fields such as request path, actor id when available, and normalized client IP metadata
- Document how operators inspect these events during an incident

#### Deliverables
- security event helper
- logging or persistence wiring
- documented event taxonomy

#### Verification
- limiter and lockout paths produce structured events
- logs contain enough fields to distinguish auth abuse from upload abuse

### Task Group 5: Edge Protection Runbook and CI Guardrails

#### Objective
Ensure app-level hardening is backed by deployment-level controls and does not silently regress.

#### Tasks
- Write a short runbook covering CDN/WAF placement, origin shielding, TLS, rate-limit policy ownership, and emergency threshold tightening
- Document which attacks are handled at edge versus in app code
- Add backend test commands or smoke checks to CI for new security helpers
- Add a checklist for production env vars and edge prerequisites before deploy

#### Deliverables
- deploy/security runbook document
- updated CI workflow or test instructions
- operator checklist

#### Verification
- CI runs the backend security-related tests
- runbook lists exact controls needed before production launch
- production checklist includes edge-layer DDoS protection explicitly

## Dependency Map

### Hard Dependencies
- baseline bootstrap must exist before route-specific limiters can be wired cleanly
- auth limiter decisions must be defined before login tests can be finalized
- upload auth must land before upload-specific threshold tuning

### Soft Dependencies
- security event design can start with logs and later expand to DB persistence
- edge runbook can be drafted in parallel once the app-side control list is stable

## Risk Register

### R-01: False-positive lockouts for shared office IPs
- Mitigation: combine IP and account-aware signals, use short lock windows first, and keep thresholds configurable

### R-02: Broken local development due to over-strict CORS
- Mitigation: keep localhost allowances in non-production and verify env parsing explicitly

### R-03: Production limiter is ineffective across instances
- Mitigation: require a shared store in production and make memory fallback development-only

### R-04: Upload restrictions break admin content workflows
- Mitigation: verify actual frontend usage first, keep admin access path intact, and test supported file types explicitly

### R-05: Teams assume app middleware solves DDoS alone
- Mitigation: make edge controls an explicit exit criterion and deployment checklist item
