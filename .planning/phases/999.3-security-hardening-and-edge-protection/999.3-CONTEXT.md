# Phase 999.3: Security Hardening & Edge Protection - Context

**Gathered:** 2026-04-21
**Status:** Ready for planning
**Source:** Manual security review of backend/frontend/deploy config plus `.planning/HANDOFF-pre-deploy.md`

<domain>
## Phase Boundary

This phase adds the minimum production security baseline needed before broader DigitalOcean rollout.

The phase covers:
- brute-force and credential-stuffing protection for authentication
- proxy-aware rate limiting for sensitive and expensive routes
- safer request parsing, body-size limits, and CORS allowlists
- upload endpoint hardening and admin-only protection
- structured security event logging and operator runbooks
- edge-layer DDoS guidance so app-layer controls are paired with infrastructure controls

The phase does **not** cover:
- a full auth redesign to refresh tokens or cookie-based sessions
- MFA rollout for all users
- database-level RLS adoption
- malware scanning or full content moderation pipelines for uploads
- vendor lock-in to one specific WAF/CDN provider

</domain>

<decisions>
## Implementation Decisions

### Threat Model
- **D-01: Treat DDoS as a layered problem**: volumetric attacks are handled at the edge/load-balancer/WAF layer, while the Node app enforces route-level rate limits, lockouts, and payload limits.
- **D-02: Protect the highest-risk routes first**: `/api/auth/login`, `/api/upload`, `/api/admin/dashboard`, `/api/admin/analytics`, and other write-heavy admin endpoints get the strictest controls first.
- **D-03: Keep login errors generic**: authentication failures must not reveal whether an email exists, whether an account is inactive, or whether a password was close to correct.

### Runtime Architecture
- **D-04: Shared limiter state in production**: production rate limiting must use a shared store such as Redis/Valkey. In-memory counters are acceptable only for local development.
- **D-05: Proxy-aware IP detection**: the backend must explicitly configure `trust proxy` and normalize client IP extraction so rate limits behave correctly behind Vercel, DigitalOcean load balancers, or reverse proxies.
- **D-06: Global parser limits should be small**: `express.json()` and `express.urlencoded()` must use conservative defaults, with large upload allowance isolated to upload middleware instead of the whole app.

### Endpoint Hardening
- **D-07: Uploads must require admin-panel authentication**: `/api/upload` is currently used only by admin management flows, so it should require `verifyToken` plus `verifyAdminPanelAccess`.
- **D-08: Validate files by content, not MIME alone**: keep allowlists, but also verify actual file signatures where possible and generate collision-safe filenames without public overwrite behavior.
- **D-09: CORS must be explicit in production**: `cors()` wide-open defaults are not acceptable for production. Use env-driven allowlists, keep localhost allowances only in non-production, and continue allowing requests with no `Origin` only when appropriate.

### Detection and Operations
- **D-10: Suspicious events must be observable**: repeated login failures, lockouts, upload denials, and limiter blocks should produce structured security events for debugging and incident response.
- **D-11: Edge controls are mandatory before launch**: deploy behind a CDN/WAF or managed reverse proxy with rate limiting, bot mitigation, TLS termination, and origin shielding. App-only defenses are insufficient for DDoS protection.

</decisions>

<guardrails>
## Critical Platform Rules (DO NOT BREAK)

- **G-01: Preserve the current login API contract**: `/api/auth/login` must keep returning `{ token, user }` inside the existing success envelope so the current webapp login flow continues working.
- **G-02: Do not break local development**: localhost origins must still work in non-production and developers must have a clear env contract for allowed origins and limiter config.
- **G-03: Avoid false-positive lockouts**: admin users and legitimate learners must be slowed or locked only by well-defined thresholds, durations, and reset rules.
- **G-04: Avoid multi-instance blind spots**: do not ship production limiter logic that silently depends on per-process memory when the app may run behind multiple instances or cold starts.
- **G-05: Upload hardening must not reopen document bypasses**: document URLs should keep following the signed-access pattern already used by the user-facing document viewer flow.

</guardrails>

<specifics>
## Specific Ideas

- Add a security bootstrap module that wires `helmet`, `cors`, `trust proxy`, request-size limits, and rate limiters in one place.
- Introduce a production shared rate-limit store (`REDIS_URL` or equivalent) rather than relying on memory.
- Add route groups for:
  - auth limiter: strict
  - upload limiter: strict + admin-only
  - admin analytics limiter: moderate
  - default API limiter: broad safety net
- Add a `SecurityEvent` persistence/logging path for repeated auth failures, lockouts, upload rejections, and rate-limit blocks.
- Create a deploy runbook for edge-layer controls, origin restrictions, alerts, and emergency response.

</specifics>
