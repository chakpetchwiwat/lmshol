# Discussion Log: Phase 999.2 - TanStack Query Migration

**Date:** 2026-04-18
**Participants:** User, the agent
**Mode:** Plan refinement from reviewed implementation document

## Summary

This phase was derived from a reviewed implementation plan for migrating frontend data fetching to TanStack Query. The original plan had the right direction, but it needed alignment with the actual codebase before execution.

## Findings From Plan Review

### F-01: Some API names in the original plan did not match the real client
- Real examples from `src/utils/api.js`:
  - `goalAPI.getGoalDetails`
  - `userAPI.getRewards`
  - `adminAPI.getDashboardStats`
  - `adminAPI.getAdvancedAnalytics`

### F-02: Query keys in the original plan were too user-centric
- The reviewed plan used `userId` in several keys by default.
- This was rejected because the current app scopes most data by auth token, not by explicit `userId` params.

### F-03: Not every request should become a query
- Signed document access and short-lived URLs should stay on-demand.
- Submit quiz and progress update should be modeled as mutations.

### F-04: `LessonPlayer.jsx` must be first
- It already had a real unstable-deps bug related to `toast`.
- It also has the highest UX sensitivity because of quiz summary state and lesson transitions.

## Agreed Planning Direction

### A-01: Use a wave-based rollout
1. Shared infra
2. `LessonPlayer.jsx`
3. Priority A user pages
4. Priority B user pages
5. Admin pages
6. Reports + cleanup

### A-02: Preserve existing UX first, clean architecture second
- The migration is successful only if it reduces bugs without breaking quiz flow or page transitions.

### A-03: Build query/mutation hooks as a reusable layer
- Components should stop owning request lifecycle details directly.
- Shared hooks should hold fetch orchestration, conditional enabling, and invalidation policy.

### A-04: Add execution-ready planning artifacts
- The todo was upgraded into a wave-based checklist.
- This phase adds a dedicated `PLAN` so implementation can start from a stable structure instead of a freeform todo.

## Open Risks To Watch During Execution

- Risk that `LessonPlayer.jsx` accidentally loses the result container after mutation refactor
- Risk that stale course detail data is shown during rapid lesson switching
- Risk that admin CRUD pages over-invalidate and refetch too broadly
- Risk that query error handling introduces duplicated toasts
