<domain>
## Phase Boundary

This phase migrates frontend server-state fetching in `elearning-webapp` from ad hoc `useEffect + useState` patterns to TanStack Query v5.

The phase covers:
- shared query infrastructure
- query key conventions
- query/mutation hook layer
- migration of user-facing pages first
- migration of admin pages after the lesson flow is stable

The phase does **not** cover:
- broad UI state management changes
- replacing local form state with server-state abstractions
- optimistic updates across the whole app
- redesigning page UX unrelated to data fetching

</domain>

<decisions>
## Implementation Decisions

### Migration Strategy
- **D-01: Wave-based migration**: Execute in waves, starting with shared infra, then `LessonPlayer.jsx`, then remaining user pages, then admin pages.
- **D-02: Guard critical learning flow first**: `LessonPlayer.jsx` is the first execution target because it has the highest regression risk and already had a real unstable-deps bug.
- **D-03: Keep local UI state local**: Modal state, active tabs, draft forms, quiz answers, and transient interaction state remain in local component state.

### Query Design
- **D-04: Resource-scoped query keys**: Query keys must be based on real resources and stable params only. Do not include `userId` in keys unless the request is actually parameterized by `userId`.
- **D-05: Shared query client**: Centralize TanStack Query setup in `src/lib/queryClient.js` and wire it once in `src/main.jsx`.
- **D-06: Signal-aware reads**: GET wrappers in `src/utils/api.js` must accept optional request config so TanStack Query can pass `signal` into axios-backed requests.

### Data Fetching Rules
- **D-07: Signed document access is on-demand**: Short-lived document access URLs are not primary cached queries. Keep them as explicit user-triggered actions.
- **D-08: Invalidate precisely**: Prefer narrow invalidation by resource key rather than broad invalidation like `['admin']`.
- **D-09: Query error handling**: Avoid toast spam from query render paths. Use `mutation.onError` for toast-first handling and UI states for query errors by default.

### Cache Policy
- **D-10: Default cache only as baseline**: Global `staleTime` is a baseline, but high-value resources like dashboard analytics, profile, settings, and lesson detail should set per-hook values when needed.

</decisions>

<guardrails>
## Critical Platform Rules (DO NOT BREAK)

### Lesson Flow
- **G-01: Result container is mandatory**: After quiz submission, `QuizSection.jsx` must still display the large result card and scroll to it.
- **G-02: Review must not reset state**: Review mode must not clear `quizResult` or `answers`.
- **G-03: No full-page flash**: During lesson transitions, `LessonPlayer.jsx` must keep the current layout visible and use a soft overlay loader instead of a full-screen reset.
- **G-04: Initial-load exception only**: The full-screen loader is allowed only when the lesson has not loaded yet.

### Migration Safety
- **G-05: No one-shot rewrite**: Do not migrate all 20 pages in one pass. Each wave must be build-verified before moving on.
- **G-06: No query-first overreach**: Do not move non-server state into TanStack Query just to “standardize” state management.
- **G-07: No broad invalidation by habit**: Invalidation must be justified per mutation and tied to real dependent resources.

</guardrails>
